# Архитектура калькулятора BitrixGPT

Карта проекта: где какая логика и как всё устроено. Обновляйте этот файл при крупных изменениях.

## Принцип

- **Один самодостаточный файл** `index.html` (HTML + CSS + JS внутри). Без бэкенда — все расчёты в браузере.
- **Две версии**:
  - **calc** (основная) — репозиторий `calc`, страница `https://belmihmed.github.io/calc/`.
  - **calc-serv** (параллельная) — репозиторий `calc-serv`, страница `https://belmihmed.github.io/calc-serv/`.
  - `calc-serv` = `calc` + фича **«Настройка от интегратора»**, контакты менеджера и партнёрский CTA в отчёте.
- Общая логика, тарифы, сценарии, тесты и документация должны оставаться синхронизированными в двух репозиториях.

## Вся логика — в `index.html`

Правки вносятся только в корневой `index.html`. Ниже — карта по секциям (ищите по имени функции/константы, а не по номерам строк — они смещаются).

| Секция (что искать в коде) | За что отвечает |
|---|---|
| `const PRICE_CONFIG = {` | Единый источник тарифов, подписок BitrixGPT, коробочных лицензий и даты актуальности цен. Из него собираются списки тарифов для модалки и расчёты отчёта. |
| `const HIDE_REVENUE`, `function revShown` | Флаг временного скрытия «Прироста выручки». `true` — скрыт, `false` — вернуть. «Потенциал базы» не затрагивает. |
| `groupFmt`, `parseNum`, `onNumInput` | Единый ввод чисел: очистка нечисловых символов, поддержка запятой/точки, разделители разрядов и сохранение позиции курсора. |
| **`const scenarios = [`** | ⭐ **Сердце проекта.** Массив всех блоков калькулятора. Каждый объект = один блок: `id`, `title`, `sub`, `link`, `role`, `effect` (`fot`/`rev`), группа (`crm`/`agents`), `fields[]` (поля с дефолтами), `compute(v)` (формула), `revLabel`, `segments`, `revOnly`-поля. **Добавить/изменить блок или формулу — здесь.** |
| `const state = {}` | Состояние сценариев: выбранность, значения полей, дополнительные сегменты и `dirty`-флаги ручных правок. |
| Рендер карточек (в `scenarios.forEach`) | Построение блоков в DOM: поля, теги, ссылка «Подробнее», кнопка «добавить сегмент». Прячет `revOnly`-поля при `HIDE_REVENUE`. При ручном вводе ставит `state[id].dirty[fieldKey] = true`. |
| `function computeScenario` | Суммирует результат по основному набору полей + всем сегментам блока. |
| `const AUTOFILL`, `function applyCompany` | Блок «Данные о компании»: какие поля компании в какие поля блоков подставлять. Автозаполнение пропускает поля, которые пользователь уже менял вручную (`dirty`). |
| `INTEGRATOR_DEFAULT`, `integratorCost` | Только `calc-serv`: дефолтные разовые стоимости настройки по сценариям и чтение пользовательских значений из `localStorage`. |
| `function recompute` | Главный пересчёт: вызывается при любом изменении. Считает суммы (ФОТ, выручка, потенциал), рисует результат каждого блока и итог. |
| `function calculateEconomics` | Единое финансовое правило для страницы и отчёта: годовая экономия ФОТ минус затраты первого года. Потенциал функция не принимает. |
| `function renderSummary` | Итоговый блок: экономия ФОТ, минимальные затраты первого года, чистая экономия и отдельно дополнительный потенциал/выручка. |
| `const INFO = {` | Тексты пояснений «как считается» (иконка «i» → всплывающее окно). Ключ = `id` сценария. |
| `function collectResults`, `function collectCompany` | Сбор данных для PDF-отчёта. |
| **`function buildReportHtml`** | ⭐ Весь PDF-отчёт: порядок «эффект → данные о компании → блоки → затраты/тариф». Логика рекомендации тарифа Bitrix24 использует `PRICE_CONFIG`; в `calc-serv` добавлены настройка интегратора и контакты менеджера. |
| `reportOverlay`, `openReport`, `closeReport` | Предпросмотр PDF-отчёта внутри iframe-оверлея. Кнопка печати вызывает `contentWindow.print()`, при ошибке — `window.print()`. |
| `modal`, `deployMode`, `renderTariffList` | Диалог выбора формата внедрения (`cloud`/`box`) и тарифа перед формированием отчёта. |
| `LOG_ENDPOINT`, `buildLogRecord`, `sendLog`, `logReport`, `logLanding` | Журналирование состава отчётов и кликов по «Подробнее» в Google Sheets через Apps Script. Если `LOG_ENDPOINT` пустой, логирование выключено. |
| `makeQrSvg` | Генерация inline-SVG QR-кодов для ссылок в PDF-отчёте. |
| `MANAGER_DEFAULT`, `mgrLoad`, `mgrSave`, `getManager` | Только `calc-serv`: контакты менеджера для отчёта, сохранение в `localStorage`, значения по умолчанию для placeholder/демо. |
| `CTA_VARIANT` | Выбирает CTA-блок PDF-отчёта. В `calc-serv` используется карточка менеджера, в `calc` — самоподключение и заявка. |
| Слайдер (`drawer`) и модалка (`infoModal`) | «Подробнее о возможностях» → боковой слайдер (свои лендинги) или новая вкладка (внешние ссылки). «i» → всплывающее окно с пояснением. |

### Как устроен один блок (пример структуры)

```js
{
  id: 9,
  title: "Голосовое управление Битрикс24",
  sub: "Описание…",
  link: "golos-bitrix24.html",     // «Подробнее» (свой лендинг → слайдер; внешний http → новая вкладка)
  role: "Сотрудники", effect: ["fot"], agents: true,   // группа «ИИ-агенты»
  fields: [
    { k:"mgr", label:"Количество сотрудников", def:1, suf:"чел.", src:"empCount" },  // src — автозаполнение из «Данные о компании»
    { k:"sal", label:"Средний оклад сотрудника", def:100000, suf:"₽/мес", src:"empSal" },
    …
  ],
  compute: v => {                  // формула. v — значения полей
    const fot = …;
    return { fot, rev:0 };         // возвращает экономию ФОТ и/или прирост выручки
  }
}
```

Группы задаются флагами на объекте: `crm:true` → «Сценарии в CRM», `agents:true` → «ИИ-агенты», иначе → «Общие сценарии».
Суффикс поля `"доля"` автоматически превращает ввод в проценты (в расчёт идёт доля).

Дополнительные свойства сценария:
- `segments:true` включает повторяемые наборы полей. Сейчас это используется для повторных продаж по базе.
- `validate(v)` возвращает объект ошибок по ключам полей. Ошибки показываются под полями, а расчёт всё равно использует защищённую формулу сценария.
- `breakdown` в результате `compute` выводит детализацию под итогом сценария. При сегментах одноимённые строки суммируются в `computeScenario`.
- `revOnly:true` на поле скрывает его при `HIDE_REVENUE = true`, если поле относится только к расчёту прироста выручки.

## Тарифы и цены

Все стоимости живут в `PRICE_CONFIG`:
- `actualOn` — дата актуальности цен;
- `cloud[].monthly` — цена тарифа Bitrix24 в месяц;
- `cloud[].subYear` — годовая подписка BitrixGPT для облака;
- `box[].licenseYear` — стоимость лицензии Битрикс24 на первый год;
- `box[].subYear` — годовая подписка BitrixGPT + Маркетплейс для коробки;
- `box[].users` — лимит пользователей коробочной редакции.
- `boxEnterpriseExtension` — цена и емкость дополнительного Enterprise-пакета на 1 000 пользователей.

Из `PRICE_CONFIG` строятся:
- `TARIFFS` и `BOX_TARIFFS` — списки облачных и коробочных редакций для модалки;
- `COST_YEAR` — минимальная годовая стоимость BitrixGPT, берётся из тарифа `basic`.

`getRecommendedCloudTariff(employees)` всегда возвращает не менее «Профессионального»: до 100 пользователей — `pro`, выше — соответствующий Enterprise. `getCloudRecommendationText` добавляет подпись про ИИ-агентов только при выбранных агентных сценариях; иначе сообщает оптимальный тариф. `calculateCloudFirstYearCosts` учитывает разницу между текущим и рекомендуемым тарифом. Для коробки `getRequiredBoxTariffForUsers` определяет требуемую редакцию по числу сотрудников, а `calculateBoxFirstYearCosts` сравнивает ее с выбранной текущей редакцией: при апгрейде считает разницу лицензий, без апгрейда оставляет текущую более старшую редакцию без доплаты.

При обновлении цен меняйте только `PRICE_CONFIG.actualOn` и нужные значения внутри `cloud`/`box`, затем запускайте `npm test`.

## Единое финансовое правило

- Годовая экономия = суммарная экономия ФОТ за месяц × 12.
- Чистая экономия первого года = годовая экономия ФОТ − все известные затраты первого года.
- На основной странице до выбора тарифа используются минимальные затраты: минимальная подписка BitrixGPT и выбранная стоимость настройки интегратора.
- В PDF после выбора варианта внедрения используются полные рассчитанные затраты: для облака тариф/доплата и подписка, для коробки доплата за лицензию при апгрейде и подписка, а также услуги интегратора.
- Потенциал базы и прирост выручки всегда показываются дополнительно и не входят в чистую экономию, окупаемость или долю затрат.

## Отчёт, тариф и CTA

Поток формирования отчёта:
1. Пользователь нажимает кнопку скачивания отчёта.
2. Открывается `modal`, где выбирается формат `deployMode`: `cloud` или `box`.
3. `renderTariffList` показывает соответствующий список `TARIFFS` или `BOX_TARIFFS`.
4. `openReport` вызывает `buildReportHtml`, записывает HTML в `reportFrame` и открывает полноэкранный предпросмотр.
5. Кнопка «Сохранить PDF / Печать» печатает содержимое iframe.

В `calc-serv` CTA отчёта — карточка менеджера. Она берёт данные из `getManager`; QR-код строится по `MANAGER.qrLink` через `makeQrSvg`.

## Логирование

`LOG_ENDPOINT` содержит URL веб-приложения Google Apps Script. Если строка пустая, логирование выключено.

Логируются два типа событий:
- `report` — формирование отчёта: версия, формат `cloud`/`box`, iframe/host, компания, данные компании, выбранные блоки, тариф, экономия и потенциал;
- `landing` — клик по «Подробнее»: версия, host, ссылка, заголовок сценария и признак внешней ссылки.

Отправка не блокирует интерфейс: сначала используется `navigator.sendBeacon`, иначе `fetch(..., { mode:"no-cors" })`; ошибки подавляются.

## Автозаполнение и ручные правки

Поля верхнего блока «Данные о компании» подставляются в сценарии через `AUTOFILL`.
Чтобы автозаполнение не перетирало ручные корректировки:
- при ручном вводе в поле сценария выставляется `state[sc.id].dirty[fieldKey] = true`;
- `applyCompany` обновляет поле только пока `dirty[fieldKey]` не выставлен.

Это правило действует для основных полей сценариев. Дополнительные сегменты независимы и автозаполнением не обновляются.

## Настройка от интегратора

Эта секция есть только в `calc-serv`:
- `INTEGRATOR_DEFAULT` хранит дефолтные разовые стоимости настройки по сценариям;
- `integratorCost(sc)` берёт значение из `localStorage`, если интегратор его менял, иначе из `INTEGRATOR_DEFAULT`;
- поля `.int-in` редактируются прямо в карточках сценариев;
- `refreshIntegratorSummary` пересчитывает итог без полной перерисовки карточек, чтобы не терять фокус в поле ввода;
- PDF-отчёт включает настройку в `costs1` и показывает выгоду первого года за вычетом настройки.

## Контакты менеджера

Блок «Ваши контакты» создаётся только в `calc-serv`, если существует `integratorCost`.

Поля менеджера:
- `name`, `role`, `company`;
- `phone`, `email`, `telegram`;
- `qrLink` — ссылка для QR-кода в отчёте.

Введённые значения сохраняются в `localStorage` по ключу `bitrixgpt_manager_v1`.
Если поле пустое, в форме показывается placeholder из `MANAGER_DEFAULT`; если вообще ничего не введено, отчёт использует пример по умолчанию.
В модалке выбора тарифа показывается напоминание, заполнены ли контакты, и кнопка «изменить» прокручивает страницу к блоку контактов.

## Тесты

Тесты находятся в `tests/` и запускаются командой:

```bash
npm test
```

`tests/helpers/extract-calculator.mjs` читает реальный `index.html`, извлекает нужные JS-фрагменты и выполняет их в VM. Проверяется не отдельная копия формул, а фактический код страницы:
- базовые константы;
- дефолтные суммы сценариев;
- суммирование дополнительных сегментов;
- валидация сценария поддержки;
- защита ручных полей от автозаполнения;
- единое правило чистой экономии ФОТ и запрет включения потенциала в окупаемость;
- единый датированный `PRICE_CONFIG`;
- границы Professional/Enterprise, доплаты при переходе и подбор коробочной лицензии по числу пользователей;
- дефолты интегратора в `calc-serv`.

GitHub Actions запускает эти тесты на `push` и `pull_request` в `main`.

## Обвязка (не логика калькулятора)

- **Лендинги** «Подробнее»: `golos-`, `podderzhka-`, `assistant-`, `zapis-`, `documents-bitrix24.html`.
- В `calc-serv` дополнительно есть настройка интегратора, контакты менеджера и партнёрский CTA, но общая расчётная логика должна совпадать с `calc`.

## Публикация

Обе версии публикуются через GitHub Pages из ветки `main`, папка `/ (root)`. После `git push origin main` GitHub Pages подхватывает изменения за 1–2 минуты.

Общие изменения нужно коммитить и пушить в оба репозитория:
- `BelMihMed/calc`
- `BelMihMed/calc-serv`

## Куда смотреть под задачу

- **Изменить блок / формулу / дефолты** → `const scenarios = [`.
- **Изменить итоговый расчёт** → `renderSummary` + `recompute`.
- **Изменить PDF-отчёт** → `buildReportHtml`.
- **Изменить тарифы или подписки** → `PRICE_CONFIG`.
- **Изменить пояснения «i»** → `const INFO` (и `INTEGRATOR_INFO` в serv).
- **Вернуть «Прирост выручки»** → `HIDE_REVENUE = false`.
- **Автозаполнение из «Данные о компании»** → `const AUTOFILL`, `applyCompany`, `state[id].dirty`.
- **Изменить стоимость настройки интегратора** → `INTEGRATOR_DEFAULT`, `integratorCost`, поля `.int-in`.
